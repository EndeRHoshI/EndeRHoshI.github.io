笔记汇总缓存

No.1 SurfaceView 重叠问题（已经归档到 《开发笔记-UI》，留作模板样式而已）

有时候我们会遇到页面内有两个 SurfaceView 的场景，比如有一个视频播放器在上方，下方是一个地图控件。正常使用时问题不大，一旦涉及到一些要设置显隐（View.VISIBLE、View.GONE）以及离开页面又回来的情形时，在某些 Android 高版本手机（测试时用的是）就会有一些异常出现，地图控件的内容会显示在视频播放器上，即使这两个控件本来的位置并不重合，这时候如果再设置一下控件的显隐或者位置（总之就是让它再发生一些布局变化刷新一下），又会恢复正常

初步猜测，经过离开页面又返回的这个过程后，控件位置会产生一些未知变化，然后两个 SurfaceView 显示会出现异常，页面内有这种情况，要注意规避，有以下的一些方案：

  1. 不能用 GONE 和 VISIBLE 控制显隐，需要使用 alpha 来控制（xml 中设置为 gone 都不行）；原理就是让空间不存在 GONE 或者 INVISIBLE 这种可能导致异常的状态。但是这样之后，点击事件不好处理，因为以前用了 GONE 和 INVISIBLE 之后，控件就点击不到了，用 alpha 控制的话，还要另外处理
  2. 想办法设置 SurfaceView 的背景色，遮盖掉底部地图不让其透出来；要注意等视频播放器出图像后又把背景色移除掉，否则背景色会一直存在导致视频内容被遮盖，这个方案缺点在于你要控制好设置背景色和移除背景色的时机，否则会有一瞬间出现底部的控件，看起来还是有些怪
  3. 用弹窗尝试改造页面；如果你的地图控件是弹出来的或者是点击后再显示的，这种方案其实是最佳的，可以把它写在另一个弹窗页面内，一方面和页面解耦，另一方面可以避免重叠问题
  4. 还有一种方案是设置 surfaceView.setZOrderOnTop(true)；这样可以让你想要放在顶层的 SurfaceView 始终位于顶层，但是要注意这样之后它就是最顶层的，没有其他控件能遮盖它了，看到有文章说再调用一下 surfaceView.setZOrderMediaOverlay(true) 可以解决，然而实际并没有用

No.2 补间动画重置问题（已经归档到 《开发笔记-UI》，留作模板样式而已）

因为公司代码框架的问题，切换 Tab 时，补间动画（也可以叫做 View 动画）没有保留在最后一帧，而是重置到第一帧了，一般情况下好像不会这样，但是现在出现了这个问题又不好改公司代码，所以换属性动画来实现，就解决了问题，这是因为属性动画是基于对象的实际属性进行修改，从而体现出动画，而补间动画只是 View 的一些动画体现，实际上 View 没有变化，所以不知道怎么就被重置了。这里简单记录一下，加深对两种动画之间的区别的认识。

按照 Android 开发者官网上的步骤，使用的版本是 2.28-alpha，与现有的 Kotlin 版本不匹配，会报错无法编译，解决方法为去到 Hilt 的官网，引入最新的版本，相似问题见这里

类注解

  1. 首先你需要使用 @HiltAndroidApp 注解一个 Application，可以是你本身的那个，也可以是另一个完全空的，总之必须要定义一个，并注册到 AndroidManifest.xml
  2. 其次是其他类注解，需要用 @AndroidEntryPoint，且如果一个 Activity 中的 Fragment 被标注了，那么这个 Activity 也要标注
  3. Hilt 注入的字段是不可以声明成 private 的，但是 protected 却可以，要知道在一个不 open 的类中，protected 基本相当于 private 的，这个可以深入了解一下是为什么
    • 延申：一个 internal 的类 A，在一个 public 且 open 的类 B 中被实例化为 a 并被声明为 protected 的，代码会标红，个人判断是因为类 B 是可以被另一个模块的类 C 继承的，这样的话，类 C 的实例化对象 c 就可以拿到父类 B 中的 a 了，这样其实就跨模块拿到了本来是 internal 的类 A 了，所以不允许这样声明（个人觉得这里的重点在于 internal 的特殊性，它是 Kotlin 特有的模块可见，和另外三个可见性修饰符有一些差别）

TransitionDrawable 使用踩坑

最近使用 TransitionDrawable 来实现两个 drawable 的渐变处理，其中一个是有阴影的 .9 图，另一个是纯色 ColorDrawable，使用过程中发现几个问题（这个 TransitionDrawable 就是两个 Drawable 叠放而已，先声明的在下，后声明的在上）

startTransition 与 reverseTransition

startTransition 意为开始过渡，从第一张图转到第二张图，reverseTransition 意为反向过渡，如果当前在第一张，则过渡到第二张，如果在第二张，则过渡到第一张，并不必须要调用 startTransition

但是仍然可以通过 startTransition 来确保当前是从第一张转到第二张的状态，避免一直不断只调用 reverseTransition 导致自己搞糊涂到底是哪个状态了，所以建议还是两种都分别在合适的时机使用

.9 与普通 Drawable 混用问题

.9 图有一个内容区域，除了内容区域之外的，暂且称为额外区域，如果你的控件是 WRAP_CONTENT 的，你的控件在设置了这个 .9 Drawable 后，内容区域就对应你的控件的实际空间大小，除此之外还会多出一块额外区域的大小

举个例子，如果你的 .9 图额外有 20dp 高度，那么最后控件除了 WRAP_CONTENT 装下内容物的高度，还会多出 20dp 的高度，一般来说也没什么问题，但是如果你使用了 TransitionDrawable，因为它是叠放的，那么第二个 Drawable 就只会填充内容区域，而不会影响到额外区域那一块

在我踩坑的这个例子里，我本意是想做一个阴影消失/显示的过渡，但是因为阴影 .9 图把我的控件撑大了，纯色 Drawable 在显示时无法把 .9 Drawable 盖住，所以表现为切换无效果，要想解决，只需要把 .9 图的拉伸区域拉满就可以了

RecyclerView 使用踩坑

今天工作时发现，某个 RecyclerView 中，如果有控件是动态 addView 并动态控制显隐的话，会出现一些异常，比如列表中都是动态 addView 的，然后有些控件是 VISIBLE 和 GONE 之间切换的，有的 View 在 GONE 之后，宽高归零，再 VISIBLE 时，宽高不会再变化，所以给人错觉设置 VISIBLE 无效，其实它是 VISIBLE 的，还有另一种情况是，如果控件没有初值，比如 TextView 没有文本，那它的宽度一开始是 0，后续再赋予文本，这个 TextView 不会被撑开，造成没有 setText 成功的假象

如何证实呢，我们可以把 TextView 赋予一个很长的初值，然后运行程序，这时发现因为初值一开始是很长的一串，TextView 被拉到很长，后续赋予正常的短一些的字符串时，设置了 WRAP_CONTENT 的 TextView 并没有自动缩到合适的宽度了

由此得出初步的结论：如果 RecyclerView item 中有一个父控件，它内部的子控件是通过动态 addView 来添加的，那它可能会表现出一些异常，解决方案是：不要用 GONE，而是用 INVISIBLE 来隐藏，同时赋予初值让它一开始就是一个比较合理的宽高，具体原因未知，初步推断和 RecyclerView 复用机制有关，因为我把复用关掉，就没有这种异常了，暂时作个记录
